* Refactored Combat Resolution (Phase 3) for true simultaneity:
    * Deployed a batching strategy for battle resolution: all battles are now calculated state-free before any pieces are modified.
    * Deferred all piece destruction and tether losses until after all battle outcomes are determined, ensuring that pieces lost in one battle can still participate in concurrent battles.
    * Centralized piece removal and tether loss processing to eliminate logic duplication and potential race conditions in board state indexes.
    * Removed redundant board state snapshots between Phase 2 and Phase 3 to optimize server performance.
    * Improved code modularity in `03-combat.ts` by separating battle identification, resolution, and result application into distinct, sequential steps.
* Implemented additional validation rules in `resolve-turn`:
    * Blocked `TETHER_ACT` for ships that were just placed in the same turn. Re-tethering is now restricted to ships already established on the board.
    * Blocked `BOMBARD_ACT` against invisible targets (cloaked Neutrinos), ensuring that bombardment requires a visible target.
* Implemented "Human-Only" game persistence logic in `resolve-turn` (Phase 5: Finalize):
    * The game now automatically concludes if no human players remain on the board.
    * This covers the scenario where all human players are eliminated, whether bots remain or if every player (bot and human) is destroyed in the same turn.
    * In these scenarios, the game ends without declaring any player as the winner.
    * This prevents infinite bot-only play cycles and ensures server resources are freed up once the human interaction has concluded.
* Refactored win condition logic for better maintainability:
