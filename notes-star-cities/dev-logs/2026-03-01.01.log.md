My overall vision for the new game_board screen is that it will cleanly use turn state, turn planned action, and turn event records much like the backend. My idea is there will be three main parts to the screen (maybe tabs):
* players tab: shows players status, and how many stars they have so far
* previous turn replay tab. uses the previous turn state and events 
* planning tab: shows current turn state and allows the user to create planned actions by clicking in the GUI

I want the architecture for managing the state of this screen to be very clean and organized. For example, the arrows etc for the planned moves should be generated directly from the list of planned actions, the player should be able to undo to pop the last item or reset to clear the list.

The game will leverage supabase realtime subscriptions: when a player status or game status changes the UI will update accordingly.

This screen will have a url containing the game id (either in the path or as a query parameter, whichever is most appropriate)

When the game status is waiting, the same screen will show a different UI instead of the game board:
* the game parameters
* list of already joined players
* buttons to add bot players (and delete bot players if still waiting)
* button to join (button to leave if still waiting)

there will be a create game screen with a form to set the game parameters (start with just the number of players, leaving the other parameters default). on create, the create game screen redirects to the game screen.

The lobby will show the list of currently waiting games (with links to their screens) and a button/link to create game. The user will also need to see a list of their own games -> the first thing they should see is their in-progress games where they haven't yet finalized their turn. I'm not really sure what's standard practice for organizing/naming pages for a multiplayer online game, so i'll take advice on this.

There were also be a profile page. (start with just setting their username.) After the user signs in, force a redirect to their profile page until they set their username. 

I welcome any feedback about this overall vision for the frontend UI.

I'm also thinking about the styling. I think I'd like to try a minimalist high contrast black background with white outlines throughout the app. And I want all corners to be sharp, no rounded edges. I want styling applied properly with a theme, and i'm open to ideas for using a flutter library to improve our ability to customize the styling. I'm also open to suggestions for a font. 

**feedback on the proposed refactoring plan and answers to clarifying questions**
On Riverpod: I don't have that much experience with Riverpod or state management in flutter so I simetimes have difficulty discerning if my coding agent is using riverpod correctly. From what i've seen, it seems to me that coding agents overuse Riverpod, and use it inappropriately. I want to use Riverpod where it's used as intended and simplifies the app. I don't want to use Riverpod for everything in the app, it should be used for specific areas only where needed. I don't want to use Riverpod in authentication. The current authentication system works well, and it has no need for an extra state layer because it leverages the Supabase Flutter SDK. The only thing I want to change in the Auth feature is probably the sign-in page styling. I don't think I want Riverpod anywhere in the router configuration. Previously my coding agent did this for me, and I discovered it. It worked, but what was done looked like an anti-pattern to me: If I remember, i think a riverpod reference was used directly in the refresh callback (not included as an input arguement but used directly in the body, and when the Riverpod reference data changed changed I think the entire router config was rebuilding to rebuild this callback function.). (currently the refreshListenable callback is being used instead). So, this isn't going to be a Riverpod app, but there can be Riverpod in the app so long as it's not the default choice. It must be used appropriately with discernment and simpler options must be considered first. For example, the Supabase SDK for flutter has many features, for example, I think it supports streams. Before turning to Riverpod, check if that feature exist in the Supabase SDK, and if the extra layer of Riverpod adds benefits its justify its use.
Also, when Riverpod code is added, it must be explained to me so that I understand what it's being used for. I've had a lot of difficulty working with coding agents in this area before and I need more knowledge of Riverpod so I can more appropriately and confidently steer the decisions.
To this end, I created an empty doc here: /home/greg/repos/star-cities/notes-star-cities/knowledge/frontend_riverpod_use.md
When you add Riverpod code, put:
* What is problem being solved and location in the app
* What are the exact Riverpod objects/feature names that are used
* Explain how these Riverpod features operate - this should be clear but detained. Not dummed down or obscured with jargon.
* Summarize alternative solutions (preferably include exact object/feature names)
