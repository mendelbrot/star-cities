## Developer Logs

## Agent Logs

* Refactored `resolve-turn` to centralize star counting logic in `TurnContext.getFactionStarCounts()`.
* Implemented new ship acquisition rules: players now receive one acquisition roll per unique star they are anchored to.
* Implemented player ranking persistence: the server now calculates and saves a sorted `player_ranking` list to the `turn_events` table at the end of each turn.
* Updated the win condition logic to use the centralized star counting method.
* Implemented the `player-bot` Supabase Edge Function with a modular architecture:
    * Created `BotContext` to manage board state and track planned actions.
    * Implemented specialized logic for ship placement, movement (Neutrino, Parallax, Eclipse), bombardment, and expansion.
    * Integrated the bot AI into the `PLANNING` phase webhook.
* Fixed a bug in `resolve-turn` where surviving pieces targeted by bombardment were still able to move. Added `wasJustBombarded` flag to `PieceTurnContext` and integrated it into movement validation.
* Refactored `resolve-turn` to handle placement collisions:
    * Implemented `pendingPlacements` in `TurnContext` to track placements before battle resolution.
    * Updated `resolveIntents` to block same-faction placement collisions and Star City anchoring on placement turns.
    * Enhanced `resolveCombat` to treat placement collisions as "entering participants" in battles, allowing movers and multiple placers to compete for the same square.
    * Ensured successful or non-conflicting placers are applied to the board index after combat.
* Verified the backend changes with `deno check` and `deno lint`.
* Refactored frontend data fetching to use a robust "REST -> Subscribe -> REST" pattern to handle websocket fluctuations and ensure eventual consistency.
* Created a reusable base class `RobustSupabaseNotifier<T, ID>` in `lib/shared/providers/robust_stream_provider.dart`.
* Updated `gameProvider`, `playersProvider`, `gameplayTurnStateProvider`, and `gameplayTurnEventsProvider` to use the new robust fetching strategy.
* Refactored lobby providers (`robustAllGamesProvider`, `robustAllPlayersProvider`) to use the robust fetching strategy for better reliability.
* Fixed an issue where `TurnStatesNotifier` and `TurnEventsNotifier` were stuck on the first turn by implementing a `postProcess` hook in `RobustSupabaseNotifier` to handle sorting and limiting of merged/real-time data.
* Verified all changes with `flutter analyze`.
* Fixed a runtime error in the frontend when switching games while a piece is selected by refactoring `gameplayUiProvider` to be a family-based `autoDispose` provider tied to `gameId`.
* Enhanced UI resilience by using `firstWhereOrNull` from `package:collection` for piece lookups in `PlanningPanel` and `GamePlanningBoard`.
* Added `collection` as an explicit dependency in `pubspec.yaml`.
* Cleaned up an unused import in `sign_in.dart` to ensure a successful `flutter analyze` run.
* Fixed a bug where Neutrinos and Star Cities could be placed adjacent to unanchored Star Cities. They now require an anchored Star City for placement, consistent with Eclipse and Parallax units (but without tether capacity checks). Updated both `resolve-turn` backend and `GamePlanningBoard` frontend.
* Enhanced the planning UI: placed pieces on the board are now dimmed (30% opacity) to indicate they are "spent" for the turn.
* Implemented placement cancellation: tapping a placed piece (either on the game board or in the tray) now removes the `PlaceAction` and returns the piece to the tray. Updated `GameBoardBase`, `GamePlanningBoard`, and `PlanningPanel`.
* Refactored `PendingActionsNotifier` to support granular action removal: `removeMovementAndBombardment`, `removeBombardment`, `removePlacement`, `removeTether`, and `removeAnchor`.
* Fixed a potential bug where deselecting a piece would cancel its re-tethering; deselection now only cancels movement and bombardment.
* Implemented "Cancel Bombard" logic in `PlanningPanel`: the bombard button now acts as a cancellation toggle if a bombardment is already planned for the selected piece.
* Restricted valid bombardment targets in `GamePlanningBoard` to only include pieces that are currently within the player's vision AND are explicitly visible (handling cloaked Neutrinos).
* Updated `GameBoardBase` to support toggling the visibility of selection dots (`showAvailableDots`).
* Configured `GamePlanningBoard` to hide selection dots during bombardment targeting, leaving only the dashed circle to represent available targets.
