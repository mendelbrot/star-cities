## Developer Logs

Last night I set up the project, did the first commit, and set up Netlify, GitHub, and Superbase.

I'll give a quick summary of the game. So this is simultaneous move turn-based online multiplayer.
The game board is a 9 by 9 grid. And it's a topological torus.
A game has two to four players.
A little bit like a cross between a space strategy game and chess.
There are around six stars on random squares of the board.
Around four different types of pieces.
One of them is called a star city, and the other ones are ships.
is a limited visibility so that the player can only see other people's ships on squares nearby their own.
There are also ship movement limits, they need to stay within a certain distance of a anchored star city.
Star cities can move and then when they are beside a star they can anchor and this allows ship movements as mentioned. (Ships tether to a star city and a star city can support up to six ships tethered to it)
Like in chess, the pieces cannot stack on top of each other strictly one piece per square.
The objective of the game is to capture all of the star cities.
A battle happens when multiple pieces try to move into the same square.
There's no economy really, but each turn a player may be given a ship or star city with a random probability, and then can place a ship on the board beside an anchored star city.
All players think about their moves at the same time, and they can undo moves as many times as they want until they finalize their decision. Then once all of the players have clicked that they are done moving, the game server will read all of the player actions and create events from these actions to determine the next state of the game.
In the database there will be a proper ledger system recording all actions and events, and the state of the board at each turn. Game replay will be a feature. And also basic animations of moves and events will be displayed each turn.

I think what want to focus on first is the frontend. I want to code making a visual mock-up of the game in a flutter widget. I want to experiment with appearance of the game UI.
I will need help with selecting an appropriate framework and UI library, and we may change these choices a few times until we get it right.
I looked at flame. However, I'm really unsure if I would benefit from using flame or not in this type of game.
And I really want the UI widgets to look fun and appropriate for the game. I don't want boring material enterprise looking UI widgets. I wonder if it would be a better approach to use a different UI widget library or to customize the material widgets, or if flame has UI widgets.
