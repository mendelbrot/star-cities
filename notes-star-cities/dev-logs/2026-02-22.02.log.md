## Developer Logs

We've done a first pass at the at the front end mockup for the game and it's going very well. 
Next, I want to formalize the front end logic a little bit more to make it more trackable and maintainable, and easier to integrate with the back end later.
I think this might mean having the current state for the turn alongside a list of actions performed by the user that turn.
We will completely take the perspective of the blue side. I'm not sure if the frontend model should include all pieces or only pieces that are visible to the blue player.
We will show the limited visibility by applying some kind of shading effect to the squares that aren't in vision range.
There's also going to be a feature where the player's home star is in the center and the board can be shifted around to have a different square in the center using arrow buttons and then it can be reset back to the home star.
This will require tracking the player's home star as well as, I think, performing some sort of a transformation mapping between the actual board and the represented board.


## Agent Logs

### Perspectives for Future Memory
- **The Torus Grid:** The 9x9 board is a topological torus. All distance calculations (Chebyshev) and pathfinding must account for edge wrapping. Currently implemented via `math.min(d, 9 - d)`.
- **Piece Identities:** 
    - **Star City:** The foundation. Must be anchored to a star to enable ship placement and tethering.
    - **Tethered Ships:** Eclipse (Circle, range 1, bombard 2) and Parallax (Triangle, range 2). Both must stay within distance 2 of their tethered city.
    - **Neutrino:** Diamond, cloaked, independent (no tether), range 1.
- **Tether Mechanics:** Max capacity of 6 ships per city. Re-tethering is instant and should be valid if the target city is anchored and has effective capacity.
- **Placement Flow:** 3-step for ships (Piece -> City -> Square); 2-step for Star Cities and Neutrinos.
- **Combat Logic:** Weighted probability based on unit strength + 50% support from neighbors. Bombardments resolve before movement and cause "Stun."

### Response to Architecture Thoughts
- **State vs. Actions:** I strongly agree with formalizing the logic into a `TurnState` + `ActionList` pattern. This mirrors the "ledger" system we discussed for the backend and makes the "Reset" functionality trivial (just clear the action list). It also simplifies potential "Undo" features.
- **Visible Pieces Model:** From a security and "Fog of War" perspective, the frontend should ideally only know about visible pieces. However, the model should be flexible enough to handle "Last Known Positions" for enemy units that slip out of vision.
- **Torus View Transformation:** Shifting the board to center on a "Home Star" is a classic torus transformation. Using a coordinate offset `(x + offset) % 9` for rendering will work perfectly and avoid complex logic in the core game model.

### Questions for Clarification
1. **Home Star:** Is the "Home Star" assigned at the start, and does it provide any specific defensive bonuses, or is it purely a navigational anchor? A: it will be assigned at the beginning of the game and it is purely navigational.
2. **Action List Transmission:** When a player clicks "Done," will we send the raw list of actions (e.g., `Move(pieceId, target)`) or the final resulting state of their pieces? A: The list of individual actions will be sent to the server in bulk. The server will create a list of events, and calculate the state at the next turn.
3. **Stun & Anchoring:** If an anchored Star City is stunned by bombardment, does it lose its ability to support tethers or placements for that turn? A: No it doesn't. But if it is un-tethered, then it looses the ability to move.
4. **Neutrino Visibility:** When a Neutrino is "revealed" via collision or support, does it stay visible for the entire next turn, or only for the resolution phase of the current turn? A: It remains visible on the next turn. Also, any ships that support a battle are visible next turn even if they're outside of the player's vision range.
